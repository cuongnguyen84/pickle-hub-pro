  1. i.
    Eyebrow strip — date · time · type

    Geist Mono caps 10.5px, letter-spacing 0.12em. Pip is 4px gray-fg-3 (not green here — green pip reserved for page-level eyebrow). Date format: 5 THÁNG 5 (VI) / MAY 5 (EN).

  2. ii.
    Status badge — verified / pending / disputed

    Verified: lime-green text + bullet, lime-glow border. Pending: amber. Disputed: red. Right-aligned on the head row, wraps under eyebrow on narrow screens.

  3. iii.
    Team row — winner is italic + green

    Player names: Instrument Serif 26px. Winner row gets italic + lime score. Loser row stays roman, fg-3 score, opacity 0.55. On hover the loser row eases up to 0.75 — readable but still hierarchically second.

  4. iv.
    @handle · DUPR sub-line

    Geist Mono 10.5px caps. The DUPR rating is the trust signal — keeps lime accent restrained for scores only, fg-2 here so the number is readable but doesn't compete.

  5. v.
    Score column — tabular game cells

    Instrument Serif italic 56px, tabular-nums, letter-spacing -0.04em. Multi-game matches stack horizontal cells separated by a 1px hairline. Right-aligned to keep ones-digits flush.

  6. vi.
    Foot — venue · view link

    Diamond glyph (◆) marks venue, link opens /san/<slug> when wired. View link sits on the right with chevron — on hover the chevron shifts +2px and color becomes lime-400.

  7. vii.
    Card surface · hover bloom

    No card border or shadow. Hairline divider 1px below. Hover bloom = horizontal lime gradient, 4% opacity center, fades to transparent at edges. Whole card is the click target; keyboard Enter on focused card opens /tran-dau/<slug>.

  1. i.
    Badge — red border, ⚠ glyph, no fill

    .mc-badge.disputed uses color: var(--tl-red), border-color: rgba(239, 68, 68, 0.3), transparent background. The ⚠ glyph replaces the bullet of verified/pending — disputed warrants a sharper signal.

  2. ii.
    Winner styling — preserved

    Disputed status flags verifiability, not the win itself. The team that scored more games still gets italic + lime score; the loser keeps roman + dim. The dispute is about whether the recorded result is accepted, not about overturning it visually.

  3. iii.
    No card-level dimming

    Earlier draft considered greying out the whole card to signal "untrusted." Rejected — a disputed match is still news in the feed, often a key conversation. Dimming would push it below verified content visually, which contradicts the editorial promise to surface what people are talking about.

  4. iv.
    Click target → /tran-dau/<slug>

    Same as verified/pending — the whole card opens the match page where the dispute thread lives. The MatchPage is responsible for surfacing the dispute reason and the resolution affordance. The feed card just signals "there is a dispute, click to see."

  5. v.
    Tooltip on the badge (build phase)

    Out of mockup scope, but the React build should add a Radix tooltip on hover/focus of the disputed badge: "Một bên đã ghi nhận khác về kết quả trận này. Mở trận để xem chi tiết." / "One side recorded a different result. Open the match for details."

  6. vi.
    Mobile rendering — same badge, abbreviated

    Frame 04 mobile shows ⚠ TRANH CHẤP on the disputed row's badge. EN mobile would render ⚠ DISPUTED full (no abbreviation needed — same length as VERIFIED).

Design rationale decisions + tradeoffs

Why this design

Tab order — Following first, Trending second

For logged-in users with follows, Following is the personal feed (their curated community); Trending is broader discovery. Following-first mirrors the patterns proven in Strava, Twitter/X, and The Athletic. It also reinforces the social-loop incentive — log in, follow, get a personalized feed — which is what Bet #1 is selling.

For anonymous viewers the Following tab is hidden entirely (Frame 05) and a single italic-serif sign-in nudge sits below the page head. Showing Following as a disabled tab would teach users an empty affordance — better to remove it.

For logged-in users with zero follows, Following remains the default tab but renders Empty State A (Frame 06) which is itself a CTA.

Card layout — single column, 720px max-width centered

Each match has substantial structure: 2 teams × (2 names + 2 DUPRs) + multi-game scores + venue + status. Two-column grids force shorter cards and break the editorial reading rhythm. A single column at ~720px is a comfortable scan width for typography-driven content, matches Strava's activity feed, and degrades gracefully to mobile without breakpoint flips.

Trending — Option B (engagement-weighted with recency fallback)

Phase 4A renders identically to Option A because the engagement signals don't exist yet (no kudos in 4A, no comments in 4B until Sprint 4C). But the SQL/RPC contract should ship now as recency_decay + (kudos_count × 3) + (comment_count × 5) so when 4B/4C plug in the multipliers, no UI rework is needed and the sort order improves automatically.

Caveat: protect against engagement gaming by capping per-user contribution to a match's engagement score. Out of scope for 4A.

Match card visual hierarchy — winner row glows, loser dims

Two-team matches need an at-a-glance "who won" signal. The winner gets italic Instrument Serif + lime-400 score, the loser stays roman with fg-3 score and 0.55 opacity. On hover the loser eases to 0.75 — readable but still hierarchically second. Score column right-aligned with tabular-nums so 11 / 7 / 9 / 8 line up cleanly across cards.

Score column — multi-game stacking

Single-game matches show one cell. Best-of-3 stacks horizontal cells separated by a 1px hairline. Right-aligned. This avoids a separate "set score" line and keeps the visual mass anchored to the right column. Tabular numerals + Instrument Serif italic 56px (desktop) / 42px (mobile) reads as scoreboard, not data.

Status badge — restraint, not theatre

Verified is the common state, so the lime treatment is muted: lime-glow background, lime border, lime text, no full-pill fill. Pending uses amber, Disputed red — both border-only with matching text. This keeps the color noise low across a vertical timeline of mostly-verified matches. See Frame 04 (Match 4) in the desktop feed and Frame 07 for the disputed-state isolated callout — winner styling is preserved, only the badge changes.

Hover affordance — chevron reveal, not redundant link

An earlier draft included a "View match / Xem trận" link in the foot row of every card. Redundant: the whole card is already a click target to /tran-dau/<slug> and a verbose mono link competes with the venue-link beside it.

Replaced with a single ::after pseudo on .mc-foot that renders → at the right edge. On desktop it's invisible at rest (opacity 0, translateX -4px) and eases in on card hover — the affordance shows up exactly when the cursor confirms intent. On touch (mobile or @media (hover:none)) the chevron stays visible at fg-3 so the affordance is discoverable without hover.

Accessibility — WCAG AA compliance

Color tokens audited against AA contrast on the #000000 background:

  • --tl-fg #fafaf7 — 19.5:1 (body)
  • --tl-fg-2 #a8a8a3 — 9.7:1 (subline copy)
  • --tl-fg-3 #888880 — 6.2:1 (eyebrow caps, dim labels) — bumped from #6b6b66 after audit
  • --tl-fg-4 #3a3a37 — 2.0:1 — decorative-only: separator glyphs (·), team-plus (+), diamond (◆), empty-mark glyphs (∅, ◌, ·), dashed borders, list-bullet dashes
  • --tl-green #a3e635 — 11.7:1 (winner score, eyebrow pip, active tab underline)

Each match card carries role="article" + a personalized aria-label summarising teams, score, venue, and date so screen readers announce the match before navigating individual link children. Demo on Match 1 of Frames 01 and 02.

Tabs are URL-controlled (?tab=following / ?tab=trending) so keyboard navigation maps to browser back-button. Active tab carries aria-selected="true" and the tablist uses role="tablist" + role="tab" on each button (build phase responsibility).

Empty states — italic poetry, mono CTA

Each empty state is a single italic-serif line that names the situation rather than apologizing for it. Một tuần yên ắng ("A quiet week") reads as editorial prose, not a 404 page. The italic mark glyph (∅, ◌, ·) at 84px sits above the title as visual anchor — different glyph per state so they're distinguishable at a glance during QA.

Bilingual sizing

EN strings run 15-20% longer than VI on average. Tab labels and eyebrow strings are sized with that in mind: "Đang theo dõi" (12 char) vs "Following" (9 char) both fit; "ĐÔI · CASUAL" (12 char) vs "DOUBLES · CASUAL" (16 char) both fit. Match-card date format is the longest delta — VI uses "5 THÁNG 5 · 19:30" (17 char), EN "MAY 5 · 7:30 PM" (15 char) — both sit on the same row without wrap on desktop. On mobile the compact form drops the month name entirely: "5/5 · 19:30".

Animation budget — one orchestrated load, then quiet

Page load: cards stagger-reveal 80ms apart with a 6-card cap (no tail animation past the fold). Pip elements pulse 2.4s cycle for the live indicator. Hover: lime gradient bloom 250ms ease, loser opacity lifts. No scroll-jank-prone scroll-triggered effects, no micro-interactions on every element. The rule: delight on entry, restraint on dwell.

Edge cases handled

  • Singles match: omits "+" plus glyph + second player
  • Long display name: truncates with ellipsis at 18ch on mobile, 28ch on desktop
  • Long venue name: truncates at 30ch with ellipsis
  • Pending status: amber badge, normal team layout
  • Disputed status: red badge with ⚠ glyph, winner styling preserved (Frame 07)
  • Multi-game scoring: 1, 2, or 3 cells stack with 1px hairline dividers
  • Anonymous: tabs hidden, sign-in nudge in fixed position above feed
  • Zero follows + Following tab: renders Empty State A as the tab content
  • Long DUPR rating ranges (e.g. 5.12+): tabular numerals keep alignment
  • Mobile DUPR: dropped from card entirely (CSS rule). Full DUPR signal lives on /nguoi-choi/:username; the feed card just needs identity at 375px.
  • Touch / no-hover devices: chevron pseudo stays visible at fg-3 instead of waiting on hover.

What's deliberately NOT here (Phase 4A scope guard)

  • No kudos button (Phase 4B)
  • No comments thread (Phase 4C)
  • No real-time match-just-finished pulse (Sprint 5)
  • No filter chips ("doubles only", "tournament only") — gate on user demand after launch
  • No avatars in the card — names + venue is enough density (DUPR added on desktop only). Avatars would force a horizontal layout shift on mobile.
  • No "who you might know" sidebar — feed page should not pretend to be Facebook

Implementation notes (for the build prompt to follow)

  • Page route: /feed + bilingual alias /bang-tin (VI)
  • Use TheLineLayout wrapper, set title bilingual, noindex={false}
  • Tabs are URL-controlled: ?tab=following / ?tab=trending for shareability + browser back-button
  • Default tab logic in useEffect: anonymous → trending; logged-in 0 follows → trending; logged-in >0 follows → following
  • Match card extracted as <FeedMatchCard /> reusable across Phase 4B/4C — accepts match prop typed against get_player_match_history RPC return shape (already has team participants + scores + venue)
  • Two new RPCs needed: get_following_feed(p_viewer_id, p_limit, p_offset) + get_trending_feed(p_limit, p_offset). Trending RPC accepts engagement multipliers as parameters so the algorithm can be tuned without redeploy.
  • Pagination: cursor-based on played_at DESC, page size 20, "Load more" button (no infinite scroll — easier to keyboard-nav and to debug)
  • Cache: React Query staleTime 30s for following, 5min for trending
