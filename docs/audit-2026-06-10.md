## 1. Những gì đã phân tích

Hệ thống là nền tảng pickleball song ngữ Việt-Anh, solo-built, gồm 5 tầng:

- **Frontend** React 18 + TS + Vite + shadcn/ui, code-split bằng `lazy`/`Suspense`, dùng React Query cho data fetching và Supabase JS client. PWA qua `vite-plugin-pwa` (SW đăng ký thủ công, bỏ qua trong WebView Capacitor).
- **Backend** Supabase: ~71 edge function (Deno), RLS + RPC. Có workaround ES256/HS256 chủ đích (`verify_jwt=false` + verify nội bộ qua `auth.getUser()`).
- **SEO/Edge** Cloudflare Pages Functions: `_middleware.ts` chặn bot → prerender HTML (cache KV `pr:v6:`), per-route render handlers, sitemap tách segment có hreflang en/vi/x-default, IndexNow.
- **Workers** riêng: `news-fetcher` (ingest RSS), `pro-tour-scraper` (HMAC auth + URL allowlist). Deploy độc lập qua `wrangler`.
- **Bên thứ ba:** Mux (livestream), Gemini (dịch EN→VI), Resend (email), Firebase FCM (push), GA4/GSC/Ahrefs.

**Luồng dữ liệu chính:** người dùng → React → Supabase (RLS/RPC) ↔ edge functions (service_role cho tác vụ backend) → bên thứ ba. Bot crawler → Cloudflare middleware → prerender từ Supabase → HTML escaped.

**Đánh giá tổng thể:** codebase được phòng thủ tốt — không có secret hardcode, không có SQL injection động, các flow dễ race (payment, match-score) đã dùng atomic UPDATE RPC + magic_token, SSR escape HTML đầy đủ, worker scraper có HMAC + allowlist + timeout. Vấn đề tập trung ở vài lỗ hổng phân quyền và một số điểm robustness/perf.

## 3. Các cải tiến đề xuất (chưa auto-apply — cần test trước khi deploy)

### A. SQL — KHÔNG auto-apply vì migration lỗi sẽ chặn pipeline deploy. SQL sẵn để chạy + test:

**A1. Guard admin cho 5 RPC analytics** (MEDIUM). Tạo migration mới, đổi sang `plpgsql` + guard `has_role`:

```sql
-- 20260610010000_admin_analytics_guard.sql
CREATE OR REPLACE FUNCTION public.get_user_stats(p_start DATE, p_end DATE)
RETURNS JSON LANGUAGE plpgsql STABLE SECURITY DEFINER SET search_path = public AS $$
BEGIN
  IF NOT public.has_role(auth.uid(), 'admin'::public.app_role) THEN
    RAISE EXCEPTION 'forbidden' USING ERRCODE = '42501';
  END IF;
  RETURN (SELECT json_build_object(
    'new_users',   (SELECT COUNT(*)::INT FROM auth.users WHERE DATE(created_at) BETWEEN p_start AND p_end),
    'total_users', (SELECT COUNT(*)::INT FROM auth.users),
    'active_users',(SELECT COUNT(DISTINCT viewer_user_id)::INT FROM view_events
                    WHERE viewer_user_id IS NOT NULL AND DATE(created_at) BETWEEN p_start AND p_end)
  ));
END; $$;
-- Lặp pattern tương tự cho get_new_users_daily (RETURN QUERY ...), get_content_stats, get_engagement_stats, get_top_content.
```
**Rủi ro khi sửa:** thấp — admin UI dùng chính tài khoản admin nên không ảnh hưởng; phải test `supabase db reset` local để chắc body plpgsql đúng trước khi push.

**A2. Siết RLS `presence_heartbeats`** (MEDIUM):
```sql
DROP POLICY IF EXISTS "presence_upsert_own" ON public.presence_heartbeats;
REVOKE ALL ON TABLE public.presence_heartbeats FROM anon;
-- chỉ cho chủ session ghi, không cho SELECT (đọc qua RPC get_online_now SECURITY DEFINER)
CREATE POLICY "presence_write_own" ON public.presence_heartbeats
  FOR INSERT TO authenticated WITH CHECK (user_id = auth.uid());
CREATE POLICY "presence_update_own" ON public.presence_heartbeats
  FOR UPDATE TO authenticated USING (user_id = auth.uid()) WITH CHECK (user_id = auth.uid());
```
**Rủi ro:** trung bình — nếu client hiện ghi presence khi chưa login (anon) sẽ hỏng; kiểm tra hook presence trước khi áp dụng.

**A3. `merge_my_ghost_by_phone` verify phone ownership** (HIGH): chuyển logic merge vào `phone-otp-verify` (đã chứng minh sở hữu phone) và `REVOKE EXECUTE ... FROM authenticated`, hoặc thêm guard kiểm tra OTP đã verify gần đây. **Rủi ro:** cần test kỹ path merge ở `SocialEventDetail.tsx`.
